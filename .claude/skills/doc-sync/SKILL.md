# Doc Sync

현재 대화(컨텍스트)에서 논의된 구현 사항, 결정 사항, 패턴, 규칙 등을 분석해
알맞은 문서 파일(CLAUDE.md / README.md / SKILL.md / AGENT.md)에 반영합니다.

## 트리거 키워드

doc-sync, ds, 문서 업데이트, 문서 정리, 문서화, CLAUDE.md 업데이트, README 업데이트,
컨텍스트 문서화, 대화 내용 정리, 지금까지 내용 문서로, 패턴 문서화, 규칙 정리

## 사용법

```
/doc-sync
/ds
/doc-sync CLAUDE.md       # 특정 파일 명시
/doc-sync README          # README 계열만 업데이트
/doc-sync skill           # 스킬 형태로 문서화
```

## Instructions

### Step 1: 컨텍스트 분석

현재 대화에서 다음 항목을 추출한다:

1. **구현된 내용** — 어떤 기능/코드가 새로 만들어졌는가
2. **결정된 패턴/규칙** — 아키텍처 결정, 코딩 컨벤션, 주의사항
3. **문제와 해결** — 발생했던 이슈와 해결 방법 (재발 방지용)
4. **반복 가능한 워크플로우** — 앞으로도 재사용할 절차나 프로세스

### Step 2: 문서 대상 판단

추출한 내용을 아래 기준으로 분류한다:

| 내용 유형                                                                                                           | 업데이트 대상                                                          |
| ------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| 전체 프로젝트 아키텍처 규칙, 코딩 컨벤션, HTTP 클라이언트 패턴, 디자인 시스템 규칙 등 프로젝트 전반에 적용되는 규칙 | `CLAUDE.md`                                                            |
| 특정 feature/entity의 구조, 사용법, 주의사항                                                                        | `src/features/{name}/README.md` 또는 `src/entities/{name}/README.md`   |
| 앞으로도 반복할 수 있는 워크플로우, 자동화 절차                                                                     | `.claude/skills/{name}/SKILL.md` (신규 or 업데이트)                    |
| 특정 역할의 에이전트 행동 정의                                                                                      | `.claude/agents/{name}/AGENT.md`                                       |
| 이번 대화에서 발견한 디버깅 패턴, 트러블슈팅                                                                        | `CLAUDE.md` > Troubleshooting 섹션 또는 별도 `docs/troubleshooting.md` |

**판단 우선순위**:

- 프로젝트 전체에 영향 → `CLAUDE.md`
- 특정 도메인에만 해당 → 해당 슬라이스의 `README.md`
- 재사용 가능한 절차 → 스킬/에이전트

### Step 3: 기존 파일 확인

업데이트 전 반드시 기존 파일을 읽어 중복/충돌 여부 확인:

- 동일한 내용이 이미 있으면 → 최신 내용으로 갱신
- 관련 섹션이 있으면 → 해당 섹션에 추가
- 아예 없으면 → 새 섹션 생성 또는 신규 파일 생성

### Step 4: 업데이트 계획 제시

파일을 수정하기 전에 반드시 사용자에게 계획을 보여준다:

```
## 문서 업데이트 계획

### 1. CLAUDE.md
- [ ] "SSE 스트리밍 패턴" 섹션 추가 (Key Patterns 하위)
- [ ] "parseSseChunk 수정 가이드" 내용 반영

### 2. src/features/chat/README.md (신규)
- [ ] useChatStream 사용법 및 주의사항
- [ ] AbortController 패턴 설명

진행할까요?
```

### Step 5: 문서 작성

사용자 승인 후 파일 작성. 아래 원칙을 따른다:

**CLAUDE.md 작성 원칙**:

- 기존 섹션 구조 유지 (임의 섹션 추가 최소화)
- 간결하게 — 구현 예시보다 패턴/규칙 위주
- 이미 있는 내용과 중복되지 않도록

**README.md 작성 원칙**:

- 해당 슬라이스의 목적, 주요 export, 사용 예시 포함
- 다른 팀원이 처음 보고도 이해할 수 있는 수준

**SKILL.md 작성 원칙**:

- skill-writer 스킬의 형식 준수
- name은 소문자 + 하이픈만, 64자 이내
- description에 트리거 키워드 포함

### Step 6: 완료 보고

업데이트된 파일 목록과 각 변경 내용을 요약해 보고한다.

## 문서 대상별 작성 가이드

### CLAUDE.md 업데이트 시

기존 섹션에 맞게 추가:

- 새 패턴 → `## Key Patterns` 하위에 새 서브섹션
- 새 규칙 → `## Code Style > Enforced Rules` 에 bullet 추가
- 새 컴포넌트/토큰 → `## Design System` 관련 섹션에 추가
- 새 기술 스택 → `## Technology Stack` 에 추가

**절대 하지 말 것**: 기존 내용 삭제, 섹션 순서 변경, 구현 세부 코드 장황하게 삽입

### feature/entity README.md 생성 시

```markdown
# {Feature/Entity 이름}

> 한 줄 설명

## 개요

이 슬라이스의 역할과 책임 설명.

## 주요 Export

| Export   | 설명 |
| -------- | ---- |
| `useXxx` | ...  |
| `XxxApi` | ...  |

## 사용 예시

\`\`\`tsx
// 코드 예시
\`\`\`

## 주의사항

- 중요한 제약이나 주의점
```

### 신규 SKILL.md 생성 시

skill-writer 스킬의 규칙을 따른다:

- `name`: 소문자 + 하이픈, 64자 이내, 디렉토리명과 일치
- `description`: 트리거 키워드 + "다음 상황에서 자동 호출" 패턴
- 150줄 이상이면 REFERENCES.md 분리

## 예시 시나리오

### 시나리오 1: 새 패턴을 CLAUDE.md에 반영

대화에서 `z.codec()` 사용법을 논의했다면:
→ `CLAUDE.md > Key Patterns > Zod Schema Pattern` 섹션 보강

### 시나리오 2: 새 feature 설명 README 생성

`features/drive`를 구현한 대화라면:
→ `src/features/drive/README.md` 신규 생성

### 시나리오 3: 반복 워크플로우를 스킬로

"Swagger → API 클래스 생성" 절차를 논의했다면:
→ `.claude/skills/swagger-gen/SKILL.md` 업데이트 또는 신규 생성

## 주의사항

- **승인 없이 파일을 수정하지 않는다** — 항상 계획을 먼저 제시
- **CLAUDE.md는 신중하게** — 전체 프로젝트 동작에 영향을 주므로 최소한의 변경만
- **기존 내용 삭제 금지** — 추가/갱신만. 삭제가 필요하면 사용자에게 명시적 확인
- **중복 확인 필수** — 기존 파일을 반드시 Read한 후 작성
